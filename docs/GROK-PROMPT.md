# Промпт для Grok 4.5 (Cursor Agent)

Скопируй текст ниже в Cursor (Agent mode, модель Grok 4.5) как первый запрос. Держи папку `docs/`
и `.cursor/rules/` в контексте. Работай по фазам, коммить после каждой.

---

```
Ты — ведущий инженер проекта HackTrainingWebapps: тренировочный полигон attack/defense для обучения
ИБ-специалистов. Твоя задача — реализовать проект ПО СУЩЕСТВУЮЩЕЙ спецификации, а не придумывать свою.

ИСТОЧНИК ИСТИНЫ (прочитай ПОЛНОСТЬЮ перед началом, это ТЗ):
- docs/SPEC.md            — архитектура, список уязвимостей (§4 app1, §5 app2), скоринг, логи, деплой
- docs/BUILD-PLAN.md      — пофазный план с критериями приёмки. Идёшь строго по фазам 0→8.
- docs/scoring.md         — модель очков, контракт API scoreboard, config.yaml
- docs/instructor-guide.md — сценарий, тайминг раундов, подсказки
- docs/forensics.md       — какие security-события и запросы обязаны работать после реализации
- docs/cursor-setup.md    — референс-репозитории и правильные паттерны фиксов
- .cursor/rules/*.mdc     — инварианты проекта (соблюдать неукоснительно)

МИССИЯ:
Построить два веб-приложения, каждое в вариантах vulnerable/ и reference/, плюс платформу скоринга,
логирование и деплой в k3s. Формат: две команды по очереди атакуют и защищают, раунд 60–90 мин.

ЖЁСТКИЕ ИНВАРИАНТЫ (нарушение = переделка):
1. Стек фиксирован: TypeScript + Express + PostgreSQL (pg) + EJS (SSR) + pino. Не заменять.
2. app1-helpdesk чинится ТОЛЬКО конфигом/зависимостями/инфрой. Прикладной src/ в vulnerable и reference
   ИДЕНТИЧЕН — различие только в значениях env/ConfigMap/package.json/Dockerfile/k8s/DB-init. Никаких
   веток `if (VULN)` в коде. diff исходников обязан быть пуст.
3. app2-billing — уязвимости OWASP Top 10 (§5 SPEC), vulnerable/src и reference/src — РАЗНЫЙ код.
4. Реализуй ТОЛЬКО уязвимости из SPEC §4–§5. Не добавляй свои, не «чини» уязвимые варианты.
5. Флаг = TRN{<32 hex>}, регэксп в packages/shared. Флаги НИКОГДА не логируются в открытом виде.
6. Секреты платформы не коммитить; учебные слабые секреты — с пометкой `# INTENTIONALLY WEAK`.
7. k3s с `--disable traefik`, ingress-nginx. NetworkPolicy: атакующая команда видит только HTTP-порт
   стенда соперника, НЕ postgres/code-server/платформу.
8. Каждый сервис: /healthz и /readyz. Все часы — UTC.

ДЛЯ КАЖДОЙ УЯЗВИМОСТИ обязательны 4 артефакта:
(1) код в vulnerable/  (2) фикс в reference/  (3) рабочий PoC в tools/attacker-scripts/
(4) раздел в reference/SOLUTION.md по шаблону: где · эксплуатация · где флаг · фикс · след в логах.
Критерий: PoC успешен на vulnerable/ и НЕ успешен на reference/; функционал не сломан (SLA-сценарий проходит).

ПОРЯДОК РАБОТЫ:
- Работай по одной фазе BUILD-PLAN за раз. В начале фазы кратко перечисли, что сделаешь и какие критерии
  приёмки закроешь. В конце фазы — прогони критерии, покажи результат, сделай атомарный git-коммит.
- Проверяй эксплойты локально через deploy/docker/compose.dev.yml до переноса в k3s.
- Если находишь противоречие в docs или тебе не хватает решения (напр. выбор библиотеки внутри
  зафиксированного стека) — СПРОСИ меня коротким списком вопросов, не выдумывай молча.
- Не пиши код вне текущей фазы «на будущее».

НАЧНИ С:
1) Подтверди, что прочитал docs/SPEC.md и docs/BUILD-PLAN.md — тезисно перескажи мне: два приложения,
   их модель различий (config-only vs code), список уязвимостей, модель скоринга. Это проверка, что ты
   понял ТЗ, а не будешь импровизировать.
2) Затем выполни Фазу 0 (каркас репозитория) и остановись на её критериях приёмки для моего подтверждения.

Отвечай по-русски, код и идентификаторы — по-английски.
```

## Промпты для последующих фаз (по одному за раз)

Когда предыдущая фаза принята:

```
Фаза принята. Переходи к Фазе N из docs/BUILD-PLAN.md. В начале перечисли план и критерии приёмки,
которые закроешь, затем реализуй, прогони критерии, покажи результат и закоммить. Соблюдай инварианты
из .cursor/rules и docs/SPEC.md. Если чего-то не хватает — спроси списком.
```

Для проверки уязвимостей (после Фаз 1–2):

```
Прогони ВСЕ PoC из tools/attacker-scripts против обоих вариантов (vulnerable и reference) каждого
приложения. Для каждой уязвимости покажи: PoC успешен на vulnerable, провален на reference, SLA-сценарий
чекера проходит на обоих. Сведи в таблицу vuln_id | vulnerable | reference | SLA. Что не сходится — чини.
```
