# Руководство инструктора — HackTrainingWebapps

Сценарий проведения тренинга: роли, тайминг, подсказки по каждой уязвимости и чек-лист судьи.
Формат — attack/defense, две команды, два раунда со сменой ролей. Правила очков — [`scoring.md`](scoring.md),
разбор атак по логам — [`forensics.md`](forensics.md).

---

## 1. Роли

- **Инструктор/судья (1 чел.):** запускает `bootstrap-cluster.sh`, следит за табло и SLA, выдаёт подсказки,
  переключает раунды (`swap-roles.sh`), ведёт финальный разбор.
- **Команда Blue (защита):** патчит назначенный ей стенд, держит SLA=up. Доступ: свой namespace (code-server,
  kubectl, git). Цель — перекрыть кражу флагов, не сломав функционал.
- **Команда Red (атака):** ищет и эксплуатирует уязвимости стенда соперника, сдаёт флаги в scoreboard.
  Доступ: только HTTP-порт чужого стенда + токен для сабмита.

За тренинг каждая команда играет оба раунда: раунд 1 — одна атакует/другая защищает, раунд 2 — наоборот.

---

## 2. Тайминг (раунд 60–90 мин)

| Этап | Время | Действия |
|------|-------|----------|
| Брифинг | −10 мин | Легенда приложения, правила, где табло, как сдавать флаги, где code-server/kubeconfig |
| **Раунд, фаза 1** | 0–15 мин | Blue: инвентаризация, чтение логов, поиск явных мисконфигов. Red: разведка, картирование эндпоинтов |
| **Раунд, фаза 2** | 15–60 мин | Активная атака/защита. Судья выдаёт подсказки по расписанию (§4) |
| **Раунд, добивание** | 60–75/90 мин | Последний пуш; табло замораживается по свистку |
| Мини-разбор | +10 мин | Быстрый взгляд на тайм-лайн Grafana: что взяли, что закрыли |
| **Смена ролей** | +5 мин | `collect-logs.sh` → `swap-roles.sh` → `reset-round.sh`; команды меняются местами |
| Раунд 2 | 60–90 мин | Симметрично |
| Финальный разбор | +30 мин | По логам: каждая атака пошагово, что помогло/не помогло защите |

Для экспресс-формата (общий тренинг 3–4 ч): раунд = 60 мин, app1 в раунде 1, app2 в раунде 2.
Для развёрнутого (полный день): раунд = 90 мин, оба приложения в каждом раунде.

**Какое приложение когда:** начинать с **app1 «Helpdesk»** (чинится конфигом, ниже порог входа), затем
**app2 «Billing»** (правка кода, OWASP). Так команды прогреваются на простом.

## 3. Подготовка стенда (чек-лист судьи, до брифинга)

- [ ] `bootstrap-cluster.sh` отработал: namespace'ы `platform`, `team-a`, `team-b` подняты.
- [ ] Grafana открывается, 5 дашбордов провижнены, логи текут (проверить любой запрос → событие в Loki).
- [ ] Scoreboard-табло доступно всем, показывает раунд 1, тик идёт.
- [ ] planter кладёт флаги (в `planted_flags` появляются строки каждый тик).
- [ ] checker показывает оба стенда `up`.
- [ ] Каждой команде выданы: URL стенда соперника, code-server + kubeconfig своего namespace, X-Team-Token.
- [ ] NetworkPolicy проверена: Red **не** достаёт postgres/code-server/платформу соперника (только HTTP-порт app).
- [ ] `reset-round.sh` протестирован на «пустом» прогоне (baseline восстанавливается < 60 c).

## 4. Подсказки по уязвимостям (эскалация)

Выдавать по расписанию, если команда застряла: **уровень 1** на 20-й минуте, **уровень 2** на 40-й, если ещё нет
прогресса. Цель — обучение, а не тупик. Полный разбор — в `apps/*/reference/SOLUTION.md`.

### app1 «Helpdesk» (config/deps)
| Vuln | Ур.1 (мягкая) | Ур.2 (прямая) |
|------|---------------|----------------|
| V1.1 JWT-секрет | «Посмотрите, чем подписан токен» | «Секрет в `.env` — угадайте его и подпишите admin-токен» |
| V1.2 Уязвимая зависимость | «Запустите `npm audit`» | «Один пакет уязвим к RCE/prototype-pollution — обновите версию» |
| V1.3 Debug/утечка | «Что отдаётся при ошибке 500?» | «Есть `/internal/debug`, выключается через env» |
| V1.4 CORS | «Проверьте заголовки CORS» | «`CORS_ORIGIN=*` с credentials — задайте конкретный origin» |
| V1.5 Дефолтные креды | «Попробуйте очевидный пароль админа» | «`admin/admin123` — смените через `SEED_ADMIN_PASSWORD`» |
| V1.6 Заголовки | «Сравните ответ с рекомендациями OWASP Secure Headers» | «helmet выключен флагом `SECURITY_HEADERS`» |

### app2 «Billing» (OWASP, код)
| Vuln | Ур.1 (мягкая) | Ур.2 (прямая) |
|------|---------------|----------------|
| V2.1 IDOR | «Смените `:id` в URL счёта» | «Нет проверки владельца ресурса в `GET /api/invoices/:id`» |
| V2.2 SQLi | «Что будет от кавычки в поиске?» | «Параметр `q` конкатенируется в SQL — параметризуйте» |
| V2.3 Stored XSS | «Куда попадает ваш комментарий и как рендерится?» | «EJS `<%- %>` без экранирования — используйте `<%= %>`/санитизацию» |
| V2.4 Mass-assignment | «Что ещё можно передать в теле обновления профиля?» | «`Object.assign` тянет `role` — введите allow-list полей» |
| V2.5 Крипто/reset | «Как хранятся пароли и как устроен reset-токен?» | «MD5 + предсказуемый токен — bcrypt + `randomBytes`» |
| V2.6 SSRF | «Куда ходит сервер при загрузке аватара по URL?» | «Нет валидации хоста — заблокируйте приватные диапазоны» |

## 5. Ведение раунда (для судьи)

- Держите открытыми: **табло scoreboard** + дашборд **Attack timeline** и **SLA/Health** в Grafana.
- Если стенд ушёл в down не по вине Blue (баг платформы) — не штрафуйте: отметьте тики как исключённые в
  `sla_samples` (флаг `excluded=true`) или дёрните `reset-round.sh`.
- Если Blue закрыл функционал (checker → mumble/down) — это их ответственность (штраф SLA заложен), но подскажите
  на разборе, где перестарались.
- Фиксируйте first-blood — хороший момент отметить команду вслух.

## 6. Смена ролей (между раундами)

```bash
./scripts/collect-logs.sh --round 1          # архив логов+аудита+git-diff патчей
./scripts/swap-roles.sh --scoreboard-url http://127.0.0.1:3020
./scripts/reset-round.sh --team a --app app2  # baseline назад (для каждого стенда)
./scripts/reset-round.sh --team b --app app2
```

PowerShell (Windows):

```powershell
.\scripts\collect-logs.ps1 -Round 1
.\scripts\swap-roles.ps1 -ScoreboardUrl http://127.0.0.1:3020
.\scripts\reset-round.ps1 -Team a -App app2
.\scripts\reset-round.ps1 -Team b -App app2
```

Подробности флагов скриптов — [`scripts/README.md`](../scripts/README.md).

`swap-roles.sh` меняет: кто attacker/defender в labels/`rounds` (через `POST /api/round/next`),
направление NetworkPolicy (attacker → defender app HTTP only), пишет `artifacts/round-state.env`.
`reset-round.sh` возвращает уязвимый baseline (vulnerable image + wipe workspace + rollout), цель < 60 c.

## 7. Финальный разбор (по логам)

Веди по `forensics.md`. Минимальный сценарий разбора на каждую значимую атаку:
1. На **Attack timeline** найти всплеск security-событий и время.
2. По `reqId` в Loki собрать цепочку запросов атаки.
3. Для SQLi — показать реальный инъектированный запрос из логов PostgreSQL (`log_statement=all`).
4. Сопоставить с сабмитом флага в `submissions` (какой `vuln_id`, во сколько).
5. Показать git-diff патча Blue: закрыли/не закрыли, почему кражи прекратились/продолжились.
6. Вывод: что сработало в защите, где была задержка реакции, что упустила атака.

Материалы для разбора берутся из `artifacts/round-<n>-<team>.tar.gz` (собирает `collect-logs.sh`).

## 8. Частые проблемы и решения

| Симптом | Причина | Действие |
|---------|---------|----------|
| Стенд не поднимается после патча Blue | синтакс/рантайм-ошибка правки | `kubectl logs`; при затыке — `reset-round.sh` (Blue теряет патч, не прогресс атаки) |
| SLA=down у всех сразу | упала платформа/postgres | проверить ns `platform`; probes перезапустят под |
| Red не может сдать флаг | истёк TTL / чужой токен / троттлинг | проверить свежесть флага и `X-Team-Token`; 429 = подождать |
| Флаги не появляются | planter не видит стенд | проверить NetworkPolicy platform→team и логи planter |
| Grafana пустая | Promtail не собирает | проверить DaemonSet Promtail и лейблы подов |
